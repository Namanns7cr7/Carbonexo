## 4. Database (Phase 3)

PostgreSQL 16. **18 tables**, all with UUID PKs (`gen_random_uuid()`), audit columns, soft delete where appropriate, and an immutable append-only credit ledger.

**Tables:** `users`, `refresh_tokens`, `user_profiles`, `emission_factors`, `activity_logs`, `bills`, `ocr_results`, `action_templates`, `user_plans`, `prompt_templates`, `recommendations`, `coach_messages`, `credit_rules`, `credit_ledger`, `rewards`, `redemptions`, `app_config`, `audit_log`.

**Key principle:** no hardcoded business values — emission factors, credit rules, rewards, prompt templates and runtime config are **data** seeded in `V3__reference_data.sql`, read by the engines at runtime. Emission-factor values were lifted from the existing frontend catalog (`lib/carbon.ts`) so the numbers match exactly.

## 5. Backend (Phases 4 / 4b)

**Stack:** Java 17 · Spring Boot 3.3.4 · Spring Security 6 · Spring Data JPA · Flyway · springdoc-openapi · Actuator + Micrometer/Prometheus · jjwt · Azure Blob SDK (for Phase 5) · Testcontainers (test).

**Foundation:**
- Layered architecture (controller → service → repository → entity) with DTOs at the boundary.
- Everything externalized to `application.yml` via `${ENV:default}` and bound to typed `AppProperties`.
- Schema owned by Flyway (`ddl-auto: validate`); Hibernate never mutates schema.
- RFC-7807 `application/problem+json` error responses; field-level validation errors; 5xx never leak internals.
- Stateless security; CORS configured for the Next.js dev origins.

**Authentication:**
- Email + password (bcrypt) register/login.
- Access token = short-lived JWT (HS256, 15 min default).
- Refresh token = opaque random string, stored only as a **SHA-256 hash**, **rotated on every use**, revocable; 14-day default TTL.
- Endpoints: `POST /api/auth/register|login|refresh|logout`.
- Swagger "Authorize" (bearer JWT) wired.

## 6. How to run (verification steps)

> None of the below has been executed in this environment yet (no JDK / Postgres present). Docker is available.

**1. Database**
```bash
docker compose -f database/docker-compose.yml up -d
# migrations run automatically when the backend boots (Flyway),
# or apply manually with the Flyway CLI (see database/README.md)
# dev demo data:
psql postgresql://ecotrack:ecotrack@localhost:5432/ecotrack -f database/seed.sql
```

**2. Backend** (needs JDK 17 + Maven, or use the Docker one-liner)
```bash
cd backend
mvn spring-boot:run
# or, without a local JDK:
docker run --rm -v "$PWD":/app -w /app maven:3-eclipse-temurin-17 mvn -q spring-boot:run
```
- API base: `http://localhost:8080`
- Swagger UI: `http://localhost:8080/swagger-ui.html`
- Health: `http://localhost:8080/actuator/health`

**3. Frontend** (unchanged)
```bash
npm install
npm run dev        # http://localhost:3000 (or next free port)
```
Frontend↔backend wiring lands in Phase 11 (`NEXT_PUBLIC_API_BASE_URL`).

**Demo login** (after `seed.sql`): `yash@ecotrack.dev` / `password` — replace the bcrypt hash before any non-local use.

## 7. Key decisions

1. **Additive only.** Frontend structure, routing, components and design tokens are preserved. Backend lives in new sibling `backend/` and `database/` trees.
2. **Maven**, single-module to start.
3. **Flyway** migrations are the executable source of truth; `schema.sql` is the readable snapshot.
4. **Provider abstractions** (OCR, AI, Blob storage) selected by config — no provider-specific code in business logic. Each will ship a local/no-op fallback so the app runs without cloud credentials.
5. **AI default** = latest Claude (`claude-opus-4-8`, with `claude-sonnet-4-6` for light/fast paths).
6. **Secrets** come from env / Azure Key Vault — never committed.
